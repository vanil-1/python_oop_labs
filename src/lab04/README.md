## Ход работы

В рамках лабораторной работы была расширена ранее разработанная система управления недвижимостью (ЛР-1 — ЛР-3).

Если ранее архитектура строилась на основе наследования классов, то в данной работе основной акцент смещён на использование **интерфейсов (ABC)** как контрактов поведения объектов.

Интерфейсы определяют не структуру объектов, а их поведение в системе.

Были введены следующие интерфейсы:

* `RentIncome` — получение дохода от аренды;
* `Reset` — сброс состояния объекта;
* `HouseType` — определение типа объекта недвижимости.

---

## Интерфейсы (ABC)

### RentIncome

Интерфейс определяет поведение объектов, способных приносить доход.

Обязательный метод:
* `get_rent_income() -> float`

Разные классы реализуют метод по-разному:
* жилые дома учитывают коммунальные расходы и жильцов;
* коммерческие объекты учитывают бизнес-нагрузку и коэффициенты спроса.

### Reset

Интерфейс определяет возможность сброса состояния объекта.

Обязательный метод:
* `reset() -> None`

Каждый класс реализует собственную логику возврата к начальному состоянию.

### HouseType

Интерфейс определяет тип объекта недвижимости.

Обязательный метод:
* `get_house_type() -> str`

Используется для определения типа объекта без использования `isinstance`. 

По большей части необходим для фильтрации объектов в зависимости от того, являются ли они коммерчискими зданиями или частными домами. Можно было реализовать фильтрацию на основе ducktyping-а, но разработанные интерфейсы используются в обоих дочерних классах, к тому же коллекция импортированна из модуля collection из пакета модулей (libs), она же используется в лабораторной работе 3. В последующих лабораторных работах это будет переработано, будет, вероятнее всего, добавлен соответствующее поведение и/или интерфейс, благодаря чему можно будет определить тип объекта на основе его поведения, а не поднимать его тип напрямую или же через интерфейс. А в 3-й лаборатной работе будет создан отдельный скрипт для коллекции, чтобы сохранить работаспосбность кода, или же фильтрация по типам будет проходить аналогично лабораторной работе 4.

## Полиморфизм через интерфейсы

В рамках работы реализован полиморфизм через интерфейсы.

Одинаковые методы вызываются у разных классов, при этом поведение зависит от реализации интерфейса.

Пример:

item.get_rent_income()
item.reset()


---

## Универсальные функции

Реализованы функции, работающие через интерфейсы:


def reset_all(items: list[Reset]) -> None:
for item in items:
item.reset()

def total_income(items: list[RentIncome]) -> float:
return sum(item.get_rent_income() for item in items)


Особенности:

* работа через абстрактные типы;
* отсутствие зависимости от конкретных классов;
* поддержка любых реализаций интерфейсов.
